**Эталон сравнения:** Claude Code Desktop (редизайн апреля 2026, вкладка Code) — [официальный анонс](https://claude.com/blog/claude-code-desktop-redesign), [справочник](https://code.claude.com/docs/en/desktop), [quickstart](https://code.claude.com/docs/en/desktop-quickstart).

## 0. TL;DR

- **Фундамент хороший.** Архитектура (супервизор pi-процессов, JSONL-фреймер, event-коалесинг, кэши метаданных, browse-режим сессий, git-центр) продумана и покрыта тестами. Ломать её не нужно — нужно достраивать.
- **Память — это в основном не GUI, а дочерние процессы.** Сам WebView укладывается в ~150–250 МБ; 1–3 ГБ набегают из pi-процессов (Node + 20 расширений), dev-серверов превью и — критично — **осиротевших процессов**: `start_kill()` убивает только прямой процесс, а его дети (MCP-серверы, vite под npm) остаются жить. Это баг №1 (§4.1).
- **Главный стратегический разрыв с Claude Code — не панели, а harness.** Claude Code «направляет» модель системным промптом с обязательным воркфлоу (todo-декомпозиция → план → правки → **авто-верификация**), а pi-app полагается на 11 сторонних расширений, которые модель вызывает по настроению. Решение — собственное расширение `pi-app-harness` + режим auto-verify через превью (§5.1).
- **UI-консистенцию чинит не «ещё одна полировка», а дизайн-токены**: сейчас в styles.css 14 произвольных размеров шрифта, 12 радиусов, 2 разных механики закрытия меню и смесь RU/EN в ярлыках (§4.2).
- **Полноэкранный старт** — одна строка `"maximized": true` в tauri.conf.json (исправлено вместе с этим документом).
- **Git приведён в порядок:** ветка `feat/redesign-ui-and-preview` полностью влита в `origin/main`, локальный main обновлён, работа продолжается на main.

## 2. Эталон: Claude Code Desktop (апрель 2026)

### 2.1 Карта фич (вкладка Code)

Источники: [анонс редизайна](https://claude.com/blog/claude-code-desktop-redesign), [справочник Desktop](https://code.claude.com/docs/en/desktop), [quickstart](https://code.claude.com/docs/en/desktop-quickstart), [MacRumors](https://www.macrumors.com/2026/04/15/anthropic-rebuilds-claude-code-desktop-app/).

**Сессии и параллельность**
- Сайдбар со **всеми активными и недавними сессиями**; фильтры по статусу/проекту/окружению, группировка по проекту.
- **Параллельные сессии в одном проекте**: каждая получает изолированную копию через **git worktrees** (`.claude/worktrees/`, настраиваемые каталог и префикс веток, `.worktreeinclude` для gitignored-файлов).
- Split-view двух сессий (Cmd+клик по сессии в сайдбаре), циклирование Ctrl+Tab.
- **Auto-archive** сессии после merge/close её PR.
- **Side chat (⌘;)** — параллельный вопрос с контекстом сессии, не засоряющий основной тред.
- Окружения: **Local / Remote (облако, живёт после закрытия приложения) / SSH** (авто-установка на удалённой машине).
- **Dispatch**: задачи с телефона порождают Code-сессии с бейджем; push-уведомление по завершении.
- «Continue in»: переезд сессии в веб/IDE с переносом контекста.

**Рабочее пространство**
- **Панели drag-and-drop**: chat, diff, browser, terminal, file, plan, tasks, subagent — любая раскладка, ресайз краёв, ⌘\ закрыть панель.
- **Встроенный терминал** (Ctrl+`) в cwd сессии, общее окружение с агентом, вкладки.
- **Файловый редактор**: клик по пути открывает файл, точечные правки, Save, предупреждение о внешних изменениях.
- Правый клик по пути: Attach as context / Open in (VS Code, Cursor, Zed) / Show in Finder / Copy path.
- **Режимы транскрипта** (Ctrl+O): Verbose / Normal / Summary.
- **Usage-кольцо**: контекст сессии + лимиты плана в одном индикаторе.
- Полный список шорткатов ⌘/.

**Диффы и PR**
- Индикатор `+12 −1` открывает **diff view** (список файлов + изменения), «rebuilt for performance on large changesets».
- **Комментарии к строкам** прямо в диффе, пакетная отправка ⌘Enter, Claude правит и показывает новый дифф.
- Кнопка **Review code**: Claude сам ревьюит дифф и оставляет inline-комментарии (только high-signal: ошибки компиляции, явные баги, безопасность).
- **Мониторинг PR**: статус-бар CI (через gh), **Auto-fix** падающих чеков, **Auto-merge** (squash) после зелёных чеков, десктоп-уведомление о завершении CI.

**Превью и браузер**
- **Browser pane** — вкладочный браузер: dev-сервер проекта + HTML/PDF/изображения/видео из проекта + внешние сайты (чистый профиль; классификаторы безопасности + allowlist доменов; Allow once / Always allow).
- **Авто-запуск dev-сервера** после правок; конфиг `.claude/launch.json` (name/runtimeExecutable/runtimeArgs/port/cwd/env/autoPort/program) — **тот же формат, что уже поддерживает pi-app**.
- **`autoVerify` (вкл по умолчанию!): после каждой правки Claude сам делает скриншоты, читает DOM/консоль, кликает, заполняет формы и чинит найденное.** Persist sessions (куки между рестартами сервера).

**Harness-фичи, видимые в UI**
- **Tasks pane**: живой список фоновой работы сессии — сабагенты, фоновые shell-команды, [dynamic workflows](https://code.claude.com/docs/en/workflows); клик — вывод в subagent-панели, можно остановить.
- **Plan pane** — отдельная панель плана.
- Режимы разрешений: Manual / Accept edits / **Plan** / **Auto (классификатор действий)** / Bypass.
- Interrupt & steer: стоп-кнопка или ввод коррекции без остановки текущего действия.
- **Scheduled tasks** (расписания) и **Routines**.
- Уведомление ОС, когда сессия закончила задачу и ты смотришь другую.
- Skills (`/`), плагины (skills+agents+hooks+MCP+LSP) с браузером маркетплейса, **Connectors** (MCP с GUI-настройкой: GitHub, Slack, Linear, Notion…).
- **Computer use** (research preview): управление приложениями/экраном с тирами доступа per-app.
- Compaction: автоматическая при заполнении окна + `/compact`.

### 2.2 Как Claude Code «направляет» модель (harness)

Это ядро жалобы №3 пользователя. Разбор по [анализу системного промпта](https://weaxsey.org/en/articles/2025-10-12/), [Piebald-AI/claude-code-system-prompts](https://github.com/Piebald-AI/claude-code-system-prompts), [Under the hood of Claude Code](https://pierce.dev/notes/under-the-hood-of-claude-code), [how-claude-code-works](https://code.claude.com/docs/en/how-claude-code-works):

1. **Однопоточный агентский цикл** — промпт + история → ответ (текст = конец, tool calls = продолжаем) → результаты инструментов → снова модель.
2. **Принудительная декомпозиция.** Системный промпт **многократно и императивно** требует использовать TodoWrite для любой нетривиальной задачи: разбить на шаги *до* любых действий, помечать шаги in_progress/completed по ходу. Это не «расширение, которое модель может вызвать», а требование верхнего уровня, подкреплённое примерами.
3. **Верификация до объявления успеха.** В todo-список принудительно попадают шаги «прогнать тесты/сборку и починить упавшее»; промпт запрещает объявлять задачу сделанной без проверки. В Desktop это усилено `autoVerify`: после каждой правки агент сам открывает превью, снимает скриншот, читает консоль и чинит.
4. **Explore → Plan → Code → Verify** как рекомендованный цикл (best practices), поддержанный режимом Plan и панелью плана.
5. **Steering как первоклассный механизм**: пользователь印 может печатать коррекции во время работы — агент читает их после текущего действия (у pi это `steer`, pi-app уже отправляет его при стриминге — механика есть, нет культуры/подсказок в UI).
6. **Специализированные сабагенты** (Explore/Plan/Task) и **dynamic workflows** — модель может написать оркестрационный скрипт под конкретную задачу; всё видно в tasks pane.
7. **Контекст-дисциплина**: CLAUDE.md (память проекта), компакция, hooks на события жизненного цикла.

**Вывод:** «постоянный результат» у Claude Code даёт не набор инструментов, а **обязательный контур: декомпозиция → видимый прогресс → автопроверка**, зашитый в системный промпт и продукт. У pi-app все кирпичи есть (rpiv-todo, plannotator, permission-system, превью, чекпоинты), но контур не замкнут: ничего не заставляет модель им пользоваться.

## 4. Аудит кодовой базы

### 4.1 Баги и дефекты (приоритизированные)

**P0 — влияют на память/стабильность (связаны с жалобой «1–3 ГБ»)**

1. **Осиротевшие дочерние процессы при kill.** `supervisor.rs:305` (`kill_by_id`) и `preview.rs:160` вызывают `child.start_kill()` — SIGKILL получает только прямой потомок. pi спавнит MCP-серверы/Node-хелперы, `npm run dev` спавнит vite → после idle-kill, вытеснения, смены режима разрешений (перезапуск агента!) или «Стоп» превью дети переезжают под launchd и продолжают есть RAM и порты. За день накапливаются сотни МБ«приложения». **Фикс:** спавнить в собственной process group (`pre_exec: setpgid(0,0)` на Unix) и убивать группу `kill(-pgid, SIGTERM)` → таймаут → SIGKILL. Это же чинит «порт занят» при рестарте превью.
2. **Жадный спавн агента.** `ChatView.tsx:1048`: открытие пустого workspace немедленно поднимает pi-процесс (сотни МБ при 20 расширениях) до первого сообщения. PLAN.md §3.2 обещал ленивый спавн. **Фикс:** спавнить в `sendPrompt`/`ensureAgent` только по первому сообщению; для загрузки списка моделей — короткоживущий `pi --list-models` (уже в ALLOWED списке `pi_cli.rs`).
3. **Зависшие RPC-промисы после смерти агента.** `store.ts`: `pending` не отклоняются на `agent-exit` → операции (fork, switch_session…) молча висят до таймаута (до 180с у rewind). **Фикс:** на `agent-exit` реджектить все `pending` с префиксом `agentId:`.

**P1 — функциональные баги**

4. **`GitBar.createPrViaAgent` без catch** (`ChatView.tsx:109`): если агент занят в другой сессии, `sendPrompt` бросает — unhandled rejection, пользователь не видит причину. Обернуть как в Composer (тост).
5. **Удаление живой сессии не убивает агента при расхождении формы пути**: `deleteSessionAction` (`store.ts:899`) сравнивает `ws.sessionPath === path` строго, хотя везде для этого есть `samePath()`. Симптом: удалили сессию — pi продолжает писать в удалённый файл.
6. **Запись settings.json на каждый нажатый символ**: `SettingsView.tsx` GeneralTab (`defaultProvider`, `defaultModel`, `theme`) вызывает `update()` в `onChange` → атомарная запись файла на каждую букву (I/O-шторм + гонка с pi, который тоже пишет settings). Дебаунс 500 мс или commit по blur.
7. **Кэш подсветки с коллизиями**: ключ `theme:lang:len:code.slice(0,80)` (`markdown.ts:87`) — два блока с одинаковым началом и длиной получат один HTML. Использовать полный хэш (djb2 по всей строке — уже есть в `msgPinId`).
8. **`stripAnsi` (`markdown.ts:5`)** — проверить, что в регэкспе действительно литеральный `\x1b` (в файле он невидим); если нет — режется легитимный текст вида `[1m`. Заменить на явный `/\x1b\[[0-9;]*m/g` + тест.
9. **Дедуп эха user-сообщений по тексту** (`reducer.ts:124`): два одинаковых сообщения подряд («да», «да») — второе не отобразится. Дедупить по (текст + отсутствие optimistic-флага), а не по вхождению в последние 6.
10. **Превью: reload по таймеру** (`PreviewView.tsx:124`): iframe перезагружается через фикс. 1200 мс после старта — если dev-сервер поднимается дольше, видим ошибку до ручного reload. Поллить порт (`probe_url` уже есть) до 200/302 и тогда монтировать iframe.
11. **`.pi/extensions/pi-permission-system/{config.json,.pi-app-mode}` затреканы в git** — локальный permission-режим разработчика попал в репозиторий (PLAN §15 добавил `.pi/` в .gitignore, но файлы уже в индексе). `git rm --cached -r .pi/`.
12. **Escape в палитре команд стирает весь ввод** (`ChatView.tsx:483`): ожидание — закрыть палитру, текст оставить.

**P2 — UX-дефекты и мелочи**

13. Дропдауны закрываются по `onMouseLeave` (ModelPicker, ModeSelector, ContextGauge, BranchPicker, PR-меню) — неустойчиво (задел мышью — закрылось); в Sidebar-меню при этом click-outside. Единая механика: click-outside + Escape.
14. GitBar обновляется только по фронту `streaming` (`ChatView.tsx:103`) — внешние правки (руками в редакторе) не отражаются. Подписаться на `sessions-changed`-подобный fs-watcher или лёгкий поллинг при фокусе окна.
15. Контекстное меню сессии/проекта открывается не у курсора (фикс. позиция строки).
16. `Sidebar.tsx:194` — `{groupId ? "›" : "›"}` бессмысленный тернарник.
17. Diff в ReviewView рендерит все строки без виртуализации — дифф на 10к+ строк подвешивает WebView (см. «rebuilt for performance» у Claude). Виртуализация + сворачивание контекста.
18. `stderrLog` агента копится в store (200 строк), но нигде не показывается — либо панель «Логи агента», либо не хранить.
19. `streaks()`/Heatmap считают дни по UTC (`toISOString`) — «текущая серия» ломается вечером; DST может дублировать день. Считать по локальной дате.
20. `⌘3` тумблерит превью, `⌘1/2/4` переключают вьюхи — несогласованная семантика; нет `⌘K`-палитры, нет `⌘/` со списком шорткатов.
21. `useRequestCountdown` по истечении шлёт `cancelled:true`, хотя pi сам авто-резолвит по таймауту — возможен двойной ответ расширению.
22. Заголовок окна/`productName` = «Pi» — конфликтует с самим pi в Dock/переключателе. Переименовать в «Pi App»/собственный бренд.
23. `csp: null` в tauri.conf.json — включить CSP (у нас dangerouslySetInnerHTML + внешние данные из сессий; DOMPurify есть, но defense-in-depth обязателен).

### 4.2 UI-инконсистентности (жалоба №1)

Собрано по `styles.css` (837 строк) и компонентам:

- **Типографика без шкалы:** font-size 10, 10.5, 11, 11.5, 12, 12.5, 13, 13.5, 14, 15, 17, 18, 20, 24 px — 14 значений, задаются точечно. Нужна шкала из 5–6 токенов (`--fs-xs…--fs-xl`).
- **Радиусы без шкалы:** 2.5, 4, 5, 6, 7, 8, 9, 10, 12, 14, 16, 20 px. Нужно 3–4 токена (`--r-s/m/l/full`).
- **Отступы ad-hoc:** padding/margin везде руками (`8px 24px 12px`, `7px 8px 7px 14px`…). Нужна spacing-шкала кратно 4.
- **Смесь языков:** «Ask permissions» / «Accept edits» / «Plan mode» (EN) рядом с «Новая сессия» (RU); «Create PR», «steer »», «затем ↳», «compact», «fetch/pull/push», «amend» вперемешку с русскими подписями. Решение: словарь строк (`src/lib/i18n.ts`) с единой политикой (предлагаю: весь UI RU, термины git/режимов — EN как имена собственные, но одинаково везде).
- **Глифы вместо иконок:** `⌁` (режим), `▾`, `⋯`, `×`, `›`, `+`, `−` — рядом с lucide-иконками. Всё перевести на иконки одного сета/размера.
- **Две механики закрытия оверлеев** (см. баг 13) и **три вида «меню»**: `.menu`, `.dropdown`, `.palette` с разными паддингами/анимациями — унифицировать в один компонент `Popover`.
- **Нативный `<select>`** (thinking level, база чекпоинтов, sort маркетплейса) рядом с кастомными дропдаунами — на macOS выглядят инородно.
- **Кнопки-строки vs кнопки-чипы vs primary** — использование смешано (напр. «compact» — голый hint-текст-кнопка в статусной строке).
- **Тосты в правом нижнем углу перекрывают статусную строку/композер** при длинных текстах; у Claude уведомления системные.
- **Пустые состояния** разной степени проработки: чат — красивый StartScreen, Review/Marketplace — просто строка `muted`.
- **Анимации:** `msg-in` на каждой карточке инструмента (transform на финальном кадре уже чинился — §PLAN 14), но нет единых длительностей/изингов-токенов; `prefers-reduced-motion` учитывается только глобальным отключением.
- **Светлая тема вторична:** `--bg-raised: #ffffff` = `--bg` (карточки сливаются с фоном), тени слишком лёгкие; тёмная явно основная.
- **Traffic-light зона** захардкожена в px с контр-масштабированием — при uiScale ≠ 1 всё ещё бывает наезд топбара (проверить на 0.7/1.6).

### 4.3 Память: откуда 1–3 ГБ (жалоба №2)

Замеряемая модель (проверить на месте через `ps -o rss -g <pgid>`):

| Компонент | Оценка RSS | Комментарий |
|---|---|---|
| WebView (UI) | 150–300 МБ | Окно рендера 80 сообщений держит DOM в норме; большие диффы Review — главный раздуватель (см. баг 17) |
| Tauri/Rust ядро | 20–40 МБ | Кэши меты/аналитики ограничены |
| **pi-процесс × processLimit(2)** | **300–800 МБ каждый** | Node + 20 расширений + MCP-серверы (context7 и т.п. — отдельные node-процессы!) |
| **Dev-сервер превью** | 200–500 МБ | vite/next + node |
| **Сироты после kill** | растёт со временем | Баг P0-1: MCP-дети и vite переживают idle-kill/«Стоп» |
| Итого | 0.7–2+ ГБ | Соответствует наблюдаемым «1–3 ГБ», причём бо́льшая часть — дочерние процессы, а не GUI |

План снижения (§5.2): process-group kill (P0-1), ленивый спавн (P0-2), processLimit=1 по умолчанию для локальных моделей, авто-стоп превью при неактивности, чистка набора расширений (AGENT-ENV.md уже посчитал: 20 → 9–11 глобально, стартовый промпт 40К → ≤15К токенов), монитор «Память агентов» в настройках (RSS по process group + кнопка kill).

### 4.4 Тест-инфраструктура (bug testing)

Есть: cargo-тесты бэкенда (сессии/git/конфиги/фреймер), e2e agent-тест, vitest для reducer/diff. Нет:
- компонентных тестов UI (React Testing Library) — регрессии вроде «кнопка молча не работает» (история с window.confirm) ловились бы;
- smoke-теста упаковки (tauri build + запуск + resolve_pi);
- теста на утечку процессов (спавн→kill→проверка pgid);
- CI (GitHub Actions: fmt+clippy+cargo test+vitest+tsc) — сейчас всё запускается руками.

## 5. Программа работ

### 5.1 П1 — Собственный harness: «направление» модели (главное, жалоба №3)

Цель: контур Claude Code (декомпозиция → план → правки → **авто-проверка**) работает **всегда**, а не «когда модель вспомнит», и виден в UI.

1. **Расширение `pi-app-harness`** (наш npm-пакет, ставится приложением автоматически, как permission-пресеты):
   - **Системный промпт-инжект** (≤2К токенов, дисциплина AGENT-ENV.md §4): императивный воркфлоу — «нетривиальная задача ⇒ todo-список до первого действия; помечай выполнение; перед завершением — верификация (тесты/сборка/превью); после компакции — перечитай todo и сформулируй состояние».
   - **Собственный todo-инструмент** (или обёртка над rpiv-todo) с событием `setWidget` — виджет плана уже рендерится в composer-зоне (`widgetbar`); добавить структурированный формат (чекбоксы, текущий шаг) вместо plain-текста.
   - **Verify-хук:** после `tool_execution_end` для edit/write — инъекция напоминания «запусти проверку»; если в проекте есть `.claude/launch.json` — предложение/автозапуск превью + браузерного инструмента.
   - **Режим Auto-verify** (паритет с `autoVerify` Claude): тумблер в превью-панели; при вкл — после каждого хода с правками приложение само (а) перезапускает/пингует dev-сервер, (б) шлёт follow-up агенту «открой ${url}, проверь консоль/DOM, почини найденное» при установленном pi-agent-browser-native. Формат launch.json уже совместим с Claude — поддержать поле `autoVerify`.
   - Конфиг per-workspace: строгость (off / remind / enforce).
2. **Кнопка «Review code»** в Review-вкладке: готовый промпт «отревьюй текущий дифф, только high-signal» → комментарии агента рендерятся в дифф (мы уже умеем комментарии→агенту; добавить обратный поток: парсить структурированный ответ в inline-карточки).
3. **Куратор окружения** (жалоба «всё стоит сторонне»): экран «Оптимизация» в настройках, реализующий AGENT-ENV.md: показать стартовую стоимость промпта (input+cacheRead первого ответа), список расширений с вердиктами (ядро/per-workspace/дубль), кнопки «применить рекомендованный набор» (`pi remove …`, правка skills-каталогов), пресеты MCP (context7, sequential-thinking) в 1 клик.
4. **Дефолты приложения**: при первом запуске предлагать «рекомендуемый профиль pi-app» — 9 core-расширений + harness + permission-пресет Ask + компакция 70–75%.
5. **Скиллы и скилл-роутер (автоподбор под задачу)** — см. [SKILLS-AUDIT.md](SKILLS-AUDIT.md): уровень 0 ✅ (2026-07-11: мусорные 18 скиллов заменены на 7 кураторских, включая 5 собственных workflow-скиллов в [agent-skills/](../agent-skills) — code-review/verify/debug/testing/ship); уровень 1 — keyword-роутер в `sendPrompt` + проактивные чипы скиллов над композером; уровень 2 — роутинг внутри `pi-app-harness` по типу текущей todo-задачи.

Результат: поведение, за которое отвечает приложение, а не удача — и оно одинаково на Claude-моделях и локальном Qwen (AGENTS.md адаптирован под особенности [Qwable-v2](https://huggingface.co/lordx64/Qwable-v2) — см. SKILLS-AUDIT §4).

### 5.2 П2 — Память и процессы (жалоба №2)

1. Process-group spawn/kill в `supervisor.rs`, `preview.rs`, `pi_cli.rs`, `app_update.rs` (P0-1) + graceful SIGTERM→SIGKILL.
2. Ленивый спавн агента (P0-2); `processLimit` по умолчанию 1 при обнаружении локального провайдера (ollama/oMLX в models.json).
3. Реджект pending на agent-exit (P0-3).
4. Виртуализация: чат (virtua/react-virtuoso, PLAN §3.1 это уже предписывал) и дифф Review.
5. Авто-стоп превью-сервера после N минут без активности панели; индикатор «сервер жив» в тулбаре.
6. Панель «Процессы» в настройках: список агентов/серверов, RSS по pgid, uptime, kill. (`list_agents` уже есть — добавить RSS и UI.)
7. Бюджеты из PLAN §9 перевести в CI-smoke (старт < 1с, RAM UI idle < 180 МБ).

### 5.3 П3 — Дизайн-система и полировка (жалоба №1)

4. **Иконки**: только lucide, размеры из шкалы (12/14/16), убрать глифы.
5. Светлая тема: поднять `--bg-raised`/тени, проверить контраст WCAG AA.

### 5.4 П4 — Паритетные фичи Claude Code

Порядок по ценности/стоимости:

1. **Уведомления ОС** (дёшево, критично): агент закончил ход/ждёт разрешения/ошибка провайдера, когда окно не в фокусе или смотрим другой workspace. Tauri notification plugin. + Dock badge с числом ожидающих.
2. **Параллельные сессии в проекте через worktrees**: ключ `chats` → `sessionKey` (cwd+sessionId) вместо cwd; «Новая параллельная сессия» создаёт worktree в `.claude/worktrees/<branch>` (совместимо с Claude), агент спавнится в worktree; в сайдбаре — статусы всех сессий (спиннер уже есть). Auto-archive после merge PR (слушаем `git_summary.branch` + gh pr status).
3. **Встроенный терминал**: xterm.js + portable-pty в Rust; cwd сессии, вкладки. (Открывает и путь к «командам руками рядом с агентом».)
4. **Файловая панель**: клик по пути в чате/диффе → CodeMirror 6 (он же нужен для JSON-конфигов по PLAN §6) с Save/предупреждением о внешних изменениях.
5. **Side chat**: отдельный короткоживущий pi-процесс с `--session` копией текущей сессии (fork_session без записи) или read-only контекст + вопрос; UI — правая панель ⌘;.
6. **PR-мониторинг**: поллинг `gh pr checks` после создания PR, статус-бар CI, кнопки Auto-fix (промпт с выводом упавшего чека) и Auto-merge.
7. **Режимы транскрипта** Verbose/Normal/Summary (фильтр рендера timeline — дёшево, у нас уже всё в items).
8. **@-упоминания файлов** с автокомплитом (индекс файлов workspace в Rust).
9. **Panes/layout**: этап 2 — после терминала/файла/side chat перейти на dockable-раскладку (golden-layout / собственная сетка).
10. **Menu bar + Quick Entry (⌥Space) + диктовка** — PLAN §5, после пунктов выше.
11. **Подписанные релизы**: tauri-bundler + notarization, канал обновлений (сейчас самосборка останется fallback'ом).

### 5.5 П5 — Инфраструктура качества

1. GitHub Actions: `cargo fmt --check`, `clippy -D warnings`, `cargo test`, `tsc --noEmit`, `vitest run`, `tauri build` (smoke).
2. React Testing Library на критичные компоненты (Composer, ExtensionUIDock, SessionRow-меню).
3. Тест утечки процессов (spawn → kill → пустой pgid).
4. Issue-теми в GitHub из баг-листа §4.1 (каждый пункт = issue с меткой P0/P1/P2).

## 7. Приложение A: политики принятия решений

- **Не переписывать движок** (супервизор/reducer/фреймер) — только точечные фиксы из §4.1; крупные фичи строить рядом.
- **Совместимость с форматами Claude Code там, где она бесплатна**: `.claude/launch.json` (уже), `.claude/worktrees/`, `autoVerify` — чтобы проекты работали в обоих инструментах.
- **Всё «умное» — в расширение pi (`pi-app-harness`), не в Rust/UI**: harness должен работать и из pi TUI, приложение лишь включает его и рисует виджеты. Это страховка от изменений RPC.
- **Каждая итерация заканчивается на main** (мержи веток — как в этот раз, без длинных расхождений).
